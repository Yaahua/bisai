# MGBIE Track-A Predictor

本技能提供专门针对 CCL 2026 MGBIE 比赛 Track-A（不微调赛道）的端到端预测与多模型集成工作流。

**核心理念：单模型存在能力瓶颈与幻觉，必须通过 RAG 动态检索、多模型集成（Ensemble）和自生成示例增强（ReverseNER）来突破零样本抽取的极限。**

## 核心工作流（冲分版）

整个预测流程分为五个阶段，目前最高得分记录为 **0.4172（并列第一）**。

### 1. 启动前准备

- 确保已克隆官方数据集仓库到 `/home/ubuntu/official_mgbie/`
- 确保已克隆比赛仓库到 `/home/ubuntu/bisai/`
- 确保已配置 `OPENAI_API_KEY` 环境变量（用于调用 `gpt-4.1-mini`、`gpt-4.1-nano` 和 `gemini-2.5-flash`）

### 2. 执行单模型批量预测（RAG 动态提示词）

放弃固定的 Few-shot 样本，改用 TF-IDF 动态检索最相似的训练集句子作为提示词（RAG）。

```bash
# 运行 v9 预测脚本（基于 RAG 动态检索，内置非法关系过滤和漏标修复）
python3.11 /home/ubuntu/bisai/提示词库/predict_track_a_v9_targeted.py
```

**建议操作：** 至少运行两个不同架构的模型（如 `gpt-4.1-mini` 和 `gemini-2.5-flash`），生成多份独立的预测结果，为后续集成做准备。

### 3. 多模型集成投票（Ensemble）

单模型极易产生独立错误，使用多数投票（Majority Voting）可以系统性消除假阳性，极大提升精确率。

```bash
# 运行三模型或四模型集成脚本
python3.11 /home/ubuntu/bisai/脚本工具/ensemble_v4.py
```

**集成策略选择：**
- **MIN_VOTES=2（三模型）**：精确率极高，是突破 0.4 的关键（得分为 0.4172）。
- **MIN_VOTES=2（四模型）**：统计特征最接近官方期望值（无关系比例 32.0% vs 期望 32.7%），预期召回率和精确率达到最佳平衡。

### 4. 提交前终检与打包

预测和集成完成后，使用打包脚本自动验证格式、边界并生成 ZIP 文件。

```bash
# 一键验证并打包
python3.11 /home/ubuntu/bisai/脚本工具/make_submit_zip.py /home/ubuntu/bisai/数据/A榜/submit_ensemble_v4a.json
```

将生成的 `.zip` 文件提交至天池 A 榜即可。

## 进阶守擂工作流（ReverseNER 自生成示例增强）

> ⚠️ **注意**：当多模型集成导致召回率断崖式下降（如无关系比例 > 50%）时，必须启用此工作流。

根据最新学术查证，简单的多数投票会错杀大量低频合法关系。为突破此瓶颈，需引入 **ReverseNER** 框架：

1. **构建伪少样本（Pseudo Few-shot）**：不再直接抽取，而是给定实体定义，让 LLM 模仿测试集句子结构，自生成包含目标实体（如 `VAR`、`TRT`）的合成句子。
2. **实体级自一致性（Entity-Level SC）**：对同一句子采样多次，按实体出现频率进行投票，而非句子级绝对匹配。
3. **Schema-based SCIR**：在预测后立即进行硬编码的规则检测（如检测到 `TRT-HAS-VAR`），触发 LLM 二次纠正。

详细规划请参阅仓库中的最高优先级文档：`分析报告/9_守擂与长远冲分规划_突破模型集成瓶颈.md`。

## 经验教训与版本演进

本技能的预测流程经历了多次质的飞跃：

1. **v1-v5 (0.3482)**：静态 Few-shot + 后处理。**教训：单模型能力触顶，后处理容易误删正确关系。**
2. **v6-v7 (0.3746)**：引入 RAG 动态提示词和代码风格 c-ICL。**教训：动态检索最相似的句子作为参考，显著降低了模型的幻觉。**
3. **v9_clean (0.3893)**：针对性修复漏标实体（如 CROP）+ 严格清洗非法三元组。**教训：删除纯粹的假阳性（FP）是稳赚不赔的提分手段。**
4. **ensemble_v3 (0.4172 🏆)**：v7 + v9_clean + Gemini 三模型多数投票。**教训：异质模型集成是 Track-A 的终极武器，能系统性消除单模型的独立错误。**
5. **ReverseNER 规划（进行中）**：**教训：单纯“堆模型”会导致召回率暴跌，必须通过前置的自生成示例（Pseudo Few-shot）来挽回被错杀的低频关系。**
