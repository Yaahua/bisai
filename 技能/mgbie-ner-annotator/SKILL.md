# MGBIE Track-A Predictor

本技能提供专门针对 CCL 2026 MGBIE 比赛 Track-A（不微调赛道）的端到端预测工作流。

**核心理念：官方数据就是 Ground Truth，一切以拟合官方标注偏好为唯一准绳，禁止任何形式的"学术纠错"。**

## 核心工作流

整个预测流程分为五个阶段，必须严格按顺序执行。

### 1. 启动前准备

- 确保已克隆官方数据集仓库到 `/home/ubuntu/official_mgbie/`
- 确保已克隆你的比赛仓库到 `/home/ubuntu/bisai_clone/`
- 确保已配置 `OPENAI_API_KEY` 环境变量（用于调用 `gpt-4.1-mini`）

### 2. 提取最优 Few-shot 样本

不要使用随机样本，必须使用脚本从官方训练集中提取覆盖目标关系（特别是漏标和易错标关系）的最优样本。

```bash
# 运行提取脚本，生成 fewshot_v3.json
python3 /home/ubuntu/bisai_clone/脚本工具/extract_fewshot_v3.py
```

**提取策略：**
- 必须覆盖 `CROP-CON-VAR`、`GENE-LOI-TRT` 等易漏标关系。
- 必须包含 `QTL-LOI-TRT` 和 `GENE-AFF-TRT` 的正确示范，防止模型过标。
- 必须包含至少 1 条**无关系**的负样本，教导模型在没有明确关系时输出空数组 `[]`。

### 3. 执行批量预测

使用提供的 Python 脚本进行自动化批量预测。该脚本内置了比赛级 Prompt、Few-shot 示例、自动重试机制和边界校验逻辑。

```bash
# 运行预测脚本（耗时约 10-15 分钟，支持断点续传）
python3 /home/ubuntu/bisai_clone/提示词库/predict_track_a_v5.py
```

**脚本特性：**
- **Temperature=0**：确保输出的确定性。
- **自动校验**：每次大模型返回 JSON 后，脚本会自动校验实体边界和关系锚点。
- **自动重试**：如果校验失败或 JSON 格式错误，脚本会自动带上错误提示重试（最多 3 次）。
- **断点续传**：如果中途中断，再次运行脚本会自动从上次断开的地方继续。

### 4. 后处理过滤（关键提分步）

大模型预测往往会产生幻觉，输出训练集中从未出现过的非法关系组合。必须通过后处理脚本将其强制删除。

```bash
# 运行后处理脚本，过滤非法关系
python3 /home/ubuntu/bisai_clone/提示词库/post_process_v4.py
```

**过滤逻辑：**
- 读取官方 `train.json`，提取所有出现过的合法三元组（如 `GENE-AFF-TRT`）。
- 遍历预测结果，删除所有不在合法三元组集合中的关系（如 `BM-USE-CROP`）。
- 这一步是纯粹的减少 FP（假阳性），只会提分不会降分。

### 5. 提交前终检与打包

预测和后处理完成后，必须运行验证脚本，确保生成的 `submit.json` 100% 符合天池的提交要求。

```bash
# 运行验证脚本
python3 /home/ubuntu/bisai_clone/提示词库/validate_submission.py \
  /home/ubuntu/bisai_clone/数据/A榜/submit_v5.json
```

如果验证通过（输出 `✓ 所有边界和格式验证通过！`），则进行打包：

```bash
# 打包为 submit.zip
cd /home/ubuntu/bisai_clone/数据/A榜/
zip submit_v5.zip submit_v5.json
```

将 `submit_v5.zip` 提交至天池 A 榜即可。

## 资源文件说明

| 文件 | 说明 | 何时使用 |
|------|------|------|
| `提示词库/official_patterns.md` | 官方 1000 条训练数据的真实统计规律与铁律 | 预测前必读，构建 Prompt 的核心依据 |
| `提示词库/prompt_template.md` | 比赛级系统提示词与 Few-shot 组装模板 | 了解脚本内部 Prompt 结构时查阅 |

| `提示词库/predict_track_a_v5.py` | 自动化批量预测脚本（含 API 调用、重试、校验） | 阶段三执行预测时使用 |
| `提示词库/post_process_v4.py` | 非法关系过滤脚本 | 阶段四后处理时使用 |
| `提示词库/validate_submission.py` | 提交文件格式与边界终检脚本 | 阶段五打包前使用 |

## 经验教训与版本演进

本技能的预测流程经历了多次迭代，以下是关键的经验教训：

1. **v1 (0.3457)**：基础 Prompt + 随机 Few-shot。表现平平，存在大量非法关系。
2. **v2 (0.3423)**：试图在 Prompt 中加入大量"不要标这个、不要标那个"的限制。**教训：模型变得过于保守，导致 Recall 暴跌，分数反而下降。**
3. **v3 (0.3482)**：回到 v1 的预测结果，仅通过后处理删除了 95 个非法关系。**教训：后处理删除确定错误的非法关系是稳赚不赔的提分手段。**
4. **v4 (未提交)**：试图在后处理中加入距离限制和关键词验证来删除过标关系。**教训：后处理很难精准判断语义，容易误删正确关系，风险极高。**
5. **v5 (最新)**：重新提取覆盖漏标模式的 Few-shot 样本，加入负样本，并叠加非法关系过滤。**教训：解决漏标和过标的根本方法是提供高质量的 Few-shot 示范，而不是靠 Prompt 规则说教。**

## 执行准则

- **放弃纠错**：不要试图用国标、行标或论文去纠正官方的标注偏好。
- **边界至上**：宁可漏标一个实体，也绝不能提交边界计算错误的实体（会导致整个文件评测失败）。
- **拥抱空白**：遇到确实没有实体或关系的句子，勇敢地输出空数组 `[]`，这符合官方 32.7% 的数据分布。
- **自动化优先**：不要手动一条条去改 JSON，全部交由脚本处理，脚本内置的校验逻辑比人眼更可靠。
