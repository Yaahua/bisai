# CCL2026-MGBIE 竞赛现状与守擂瓶颈分析报告

## 1. 竞赛现状概述

目前，您在天池 CCL2026-MGBIE 杂粮育种信息抽取评测（Track-A 不微调赛道）中取得了 **0.4172** 的优异成绩，位列**A榜并列第一名**。这一成绩的取得，主要归功于多模型集成（Ensemble）策略的成功应用。

在经历了最初的“学术纠错”方向偏差后，您果断调整策略，严格拟合官方标注偏好。通过引入 RAG 动态提示词、c-ICL 代码风格对比学习，以及最终的 `ensemble_v3`（GPT-4.1-mini v7、GPT-4.1-mini v9_clean、Gemini-2.5-flash v10 三模型多数投票，`MIN_VOTES=2`），您成功地将精确率提升到了极高的水平，消除了大量假阳性（FP）错误。

最新提交的 `v16_rules` 版本进一步针对 `ensemble_v3` 召回率偏低的问题，引入了纯规则后处理。该版本在零 API 消耗的情况下，通过触发词匹配，成功为结果补充了 238 条高置信度关系。这使得关系均值从 2.16 回升至 2.75，无关系比例从 38.2% 降至 30.2%，完美拟合了官方训练集的统计分布（期望关系均值 2.80，无关系比例 32.7%）。

## 2. 守擂计划与长远冲分规划

在登顶之后，您制定了详尽的守擂与长远冲分规划，旨在突破“堆模型”策略带来的边际收益递减和召回率断崖式下降问题。基于对最新学术文献的查证，规划主要分为三个阶段：

| 阶段 | 核心策略 | 预期目标 | 优先级 |
| :--- | :--- | :--- | :--- |
| **阶段一** | **ReverseNER 自生成示例增强** | 解决无关系比例过高问题，提升低频关系的召回率。 | 🔴 高 |
| **阶段二** | **Schema-based SCIR 自纠正** | 基于硬规则检测非法三元组和漏标，触发 LLM 二次纠正，消除 100% 非法关系 FP。 | 🟡 中 |
| **阶段三** | **农业垂直 KDR-Agent** | 部署多智能体框架，接入农业垂直数据库，解决复杂长句中生僻实体的抽取。 | 🟢 低（备战 B 榜） |

此外，为了应对 API 成本和限速问题，您还提出了《低成本多源 API 与小模型混合冲分方案》（文档 11），计划引入免费的 GLM-4.7-Flash 作为投票成员，并在本地部署 GLiNER 和 GLiREL 等零样本小模型，结合 SuperICL 框架进行混合推理，以期在降低成本的同时提升效果。

## 3. 当前核心瓶颈分析

综合分析您的代码仓库、试错日志和规划文档，您当前面临的核心瓶颈主要集中在以下三个方面：

### 3.1 API 成本高昂与稳定性危机

这是当前最紧迫的物理瓶颈。多模型集成策略本质上是“用钱买分数”，随着集成模型数量的增加（如四模型集成），API 调用量呈线性增长。在尝试运行 v14 补充召回率时，API 积分耗尽导致任务中断（153 条中仅完成 56 条）。

同时，新增模型（如 Gemini、GPT-4.1-nano）在并发预测时容易触发 429 限速错误，导致数据包缺失。修复这些失败样本（如 `fix_failed_items.py` 脚本所示）需要耗费大量时间和额外的 API 调用，严重拖慢了迭代节奏。

### 3.2 多数投票机制导致的召回率断崖

多模型多数投票（Majority Voting）是您登顶的关键，但也是当前提升分数的最大技术阻碍。`ensemble_v3.py` 脚本的实现机制要求关系只有在头尾实体都进入合并实体列表时才被保留。这种严格的交集策略虽然极大地提升了精确率，但导致大量合法的低频关系被丢弃。

数据表明，`ensemble_v3` 的关系均值仅为 2.16（期望 2.80），无关系比例高达 38.2%（期望 32.7%）。虽然 `v16_rules` 通过纯规则后处理部分缓解了这一问题，但这仅仅是治标不治本。规则覆盖面有限，且高度依赖特定的英文触发词，无法从根本上解决模型本身对复杂关系抽取能力不足的问题。

### 3.3 零样本约束下的技术落地难度

守擂计划中提出的学术前沿方案（如 ReverseNER、GLiREL）在理论上非常完美，但在实际落地 Track-A（零样本、无微调）时面临巨大挑战。

例如，ReverseNER 依赖大模型生成合成句子作为 Few-shot，这依然需要消耗大量 API，与降低成本的目标相悖。而 GLiREL 虽然是零样本关系抽取模型，但在本地沙盒环境中的部署和推理速度、以及在农业垂直领域的实际表现仍是未知数。将这些复杂的技术整合到现有的流水线中，需要极高的工程能力和大量的试错成本。

## 4. 总结与建议

您目前的策略已经将大模型的 In-Context Learning 能力压榨到了极限，通过工程手段（集成 + 规则）取得了卓越的成绩。但要继续拉开差距，必须摆脱对昂贵 API 的依赖，转向更精细化、本地化的推理架构。

建议您优先推进《低成本多源 API 与小模型混合冲分方案》中的本地小模型（GLiNER + GLiREL）部署。这不仅能彻底解决 API 成本和限速问题，还能通过 SuperICL 框架为大模型提供高质量的先验知识，从而在保证精确率的同时，大幅提升召回率。这是突破当前瓶颈、备战 B 榜的最优路径。
