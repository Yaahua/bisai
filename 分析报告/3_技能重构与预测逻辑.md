# MGBIE 比赛技能重构方案：从“学术纠错”转向“比赛打榜”

## 1. 现有技能的根本问题

经过审查，你目前的两个技能（`mgbie-ner-annotator` 和 `mgbie-review-agent`）设计得非常严谨，甚至可以说是“教科书级别”的学术数据清洗工作流。它们包含了知识库加载、单线程防呆、对抗性审查、双重验证等高级机制。

**但是，它们完全不适合当前的比赛目标。**

这两个技能的核心逻辑是：**“发现官方数据不符合学术规范，我要把它改对。”**
而比赛的核心逻辑是：**“官方数据就是 Ground Truth，我要让模型学会官方的偏好。”**

具体来说，现有技能存在以下致命冲突：
1. **过度纠错导致标准偏移**：技能中大量引用了国标（GB）、行标（NY/T）和论文，强行纠正了官方数据中高频出现的 `LOI(QTL, TRT)` 和 `AFF(GENE, TRT)` 等关系。这直接导致了你在 A 榜预测中 RE（关系抽取）分数崩盘。
2. **流程过于繁重**：为了修改训练集，你设计了“初标 -> 撰写说明 -> 对抗审查 -> 全量修正”的漫长闭环。但实际上，**打 Track-A 根本不需要修改训练集**，只需要从中提取 Few-shot 示例即可。
3. **目标错位**：技能的目标是“输出符合比赛要求的 JSON 格式数据及过程文件”，但比赛真正需要的是“输出对 `test_A.json` 的预测结果”。

## 2. 技能重构决策：删一改一

为了以比赛得分为唯一准绳，我建议采取**“删一改一”**的重构策略：

### 决策一：彻底删除 `mgbie-review-agent`
*   **原因**：对抗性审查的目的是为了确保对训练集的修改“有理有据”。既然我们决定**不再修改官方训练集**，这个审查技能就完全失去了存在的意义。保留它只会增加维护成本和误导未来的操作。
*   **操作**：直接删除 `/home/ubuntu/skills/mgbie-review-agent` 整个目录。

### 决策二：将 `mgbie-ner-annotator` 彻底改写为 `mgbie-track-a-predictor`
*   **原因**：我们需要一个专门针对 Track-A（不微调赛道）的预测技能。它不需要修改数据，而是专注于**构建高质量的 Prompt 并调用大模型进行预测**。
*   **新技能定位**：读取官方原始 `train.json` 提取规律和示例，读取 `test_A.json`，利用大模型上下文能力（In-Context Learning）生成预测结果，并打包为 `submit.zip`。

## 3. 新技能 `mgbie-track-a-predictor` 设计方案

新技能的工作流将彻底抛弃“审查与修正”，转而聚焦于“特征提取与精准预测”。

### 核心工作流设计

1.  **阶段一：官方规律提取与 Few-shot 准备**
    *   **动作**：读取官方原始 `train.json`。
    *   **规则约束**：强制技能认知官方的特殊偏好（如 `LOI` 关系中 tail 是 `TRT` 的比例高达 53%）。
    *   **示例挑选**：自动从 `train.json` 中挑选 5-8 条包含高频实体（TRT, GENE, CROP）和高频关系（AFF, LOI, HAS）的典型样本，作为 Few-shot 示例。

2.  **阶段二：构建比赛级 Prompt**
    *   **动作**：组装系统提示词。
    *   **Prompt 结构**：
        *   **角色设定**：杂粮育种信息抽取专家。
        *   **官方铁律**：明确列出从数据分析中得出的官方标注偏好（禁止 AI 自行发挥学术纠错）。
        *   **格式要求**：严格的 JSON 输出格式。
        *   **Few-shot 示例**：阶段一提取的真实官方样本。

3.  **阶段三：批量预测与防呆校验**
    *   **动作**：读取 `test_A.json`（400条），循环调用大模型 API 进行预测。
    *   **防呆机制**：
        *   校验生成的 JSON 结构是否合法。
        *   校验实体边界（`text[start:end] == entity['text']`）。
        *   校验关系锚点（head/tail 必须对应存在的实体）。
        *   **若校验失败，自动触发单条重试**。

4.  **阶段四：打包提交**
    *   **动作**：将 400 条预测结果合并为 `submit.json`，并压缩为 `submit.zip`，准备提交天池。

## 4. 仓库清理与执行计划

为了实施这个方案，我们需要对你的 `bisai` 仓库进行一次大扫除。

### 第一步：清理历史包袱
1.  删除 `bisai` 仓库中所有由 AI 生成的“修正数据”、“审查文档”和“问题清单”（即 `数据/训练集/修订结果/`、`reviews/`、`audits/` 等目录）。
2.  删除 `mgbie-review-agent` 技能。

### 第二步：重写技能文档
1.  使用 `skill-creator` 技能的规范，将 `mgbie-ner-annotator` 的 `SKILL.md` 彻底重写为上述的 `mgbie-track-a-predictor` 逻辑。
2.  更新技能的 `description`，明确其用于 Track-A 预测。

### 第三步：编写配套脚本
1.  在技能目录下编写 `predict_track_a.py` 脚本，封装调用大模型 API、组装 Prompt 和并发预测的逻辑。
2.  编写 `validate_submission.py` 脚本，确保最终生成的 `submit.json` 完全符合天池的提交要求。
