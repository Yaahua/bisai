# CCL2026-MGBIE 比赛数据与策略分析报告

## 1. 比赛背景与赛道说明

根据天池比赛官方页面 [1] 和官方 GitHub 仓库 [2] 的说明，CCL2026-MGBIE（杂粮育种信息抽取评测）包含两个子任务：命名实体识别（NER）和关系抽取（RE）。

比赛分为两个独立的赛道：
*   **Track-A（不微调赛道）**：不可对模型参数进行微调，主要考察模型的上下文学习（In-Context Learning）能力，不限制模型参数规模。
*   **Track-B（微调赛道）**：允许在官方提供的数据集上微调模型参数，但限制模型参数规模不超过 7B。

**关于你的疑问：**
你提到“是不是我先用模型上下文能力进行训练，随后打那个a榜”。
实际上，**Track-A（不微调赛道）是不需要（也不允许）训练/微调模型参数的**。在 Track-A 中，你应该直接使用强大的大语言模型（如 GPT-4、Qwen-Max 等），通过精心设计的 Prompt（提示词）和 Few-shot（少样本示例，即上下文能力），让模型直接对测试集（A榜测试集 `test_A.json`）进行预测。

如果你想“训练”模型（即改变模型权重），你需要参加 **Track-B（微调赛道）**，并且只能使用 7B 及以下规模的模型（如 Qwen-7B、Llama3-8B 等）。

## 2. 官方训练集与你的 bisai 仓库数据对比分析

你担心“AI有没有乱搞”以及“有没有偏移”。我对比了官方原始训练集（1000条）和你 `bisai` 仓库中已经由 AI 修正过的前 300 条数据（chunk_001 到 chunk_030）。

### 2.1 数据结构与内容一致性

官方提供的 `train.json` 确实是**有标注的**，包含了 `text`、`entities` 和 `relations` 字段。
你的 AI 助手在处理时，是在官方原始标注的基础上进行“审查和修正”，而不是从零开始盲目标注。

但是，在一致性检查中发现了**轻微的文本偏移问题**：
在你的前 300 条修正数据中，有 **14 条数据的 `text` 字段被 AI 篡改了**（与官方原始文本不一致）。
例如，在 `chunk_001_corrected.json` 中，AI 生成了类似 "The waxy gene is crucial for starch synthesis..." 的文本，而这在官方原始训练集中是不存在的。这通常是因为大模型在处理长 JSON 时产生了“幻觉”（Hallucination），自行编造了文本。

### 2.2 实体与关系边界偏移

在你的前 300 条修正数据中，存在一定比例的边界偏移（Span Offset）错误：
*   **实体边界错误**：在 2036 个实体中，有 108 个实体的 `start` 和 `end` 索引切片出来的文本与 `text` 字段不匹配。例如，AI 标出的实体是 "WGCNA"，但按索引切片出来的是 "EGs. "。
*   **关系锚点错误**：在 849 条关系中，有 54 条关系的 `head_start`/`head_end` 或 `tail_start`/`tail_end` 索引与实际文本不匹配。

这些边界错误会导致在最终评测脚本计算 F1 分数时，这些实体和关系被直接判定为错误（False Positive），严重影响得分。

### 2.3 标签分布的偏移

你的 AI 助手对原始标注进行了大量的“纠正”，导致标签分布发生了显著变化：

| 标签类型 | 官方原始比例 | 你的修正后比例 | 变化趋势 |
| :--- | :--- | :--- | :--- |
| **TRT (性状)** | 23.9% | 30.4% | 显著增加 |
| **AFF (影响关系)** | 30.2% | 43.1% | 大幅增加 |
| **LOI (定位于)** | 25.5% | 20.6% | 显著减少 |

根据你的 `validation_report.md` [3]，AI 认为原始数据中存在大量不符合规范的标注（例如将“drought tolerance QTL”作为一个整体，AI 将其拆分为 TRT 和 QTL；或者原始数据中大量 LOI 关系的 tail 是 TRT/GENE 而非 CHR/QTL，AI 将其修正为 AFF）。

**这里的核心风险在于：**
如果官方测试集（A榜/B榜）的标注标准与官方训练集保持一致（即包含那些“不规范”的标注），那么你的 AI 助手所做的“完美纠正”反而会成为一种**负向偏移**。模型学习了你修正后的“完美标准”，在面对官方测试集时，预测结果会与官方的 Ground Truth 产生巨大差异，导致分数暴跌。

## 3. A榜测试集现状

我检查了官方仓库中的 `test_A.json`（共 400 条）。
目前，该文件**仅包含 `text` 字段，`entities` 和 `relations` 字段均为空列表 `[]`**。
这是正常的比赛流程：官方只提供测试集的文本，你需要用你的模型/方法预测出实体和关系，填入 JSON 中，然后打包成 `submit.zip` 提交到天池平台，平台会计算你的得分。

## 4. 总结与后续建议

### 4.1 关于“AI有没有乱搞”
AI 确实在认真执行你设定的“审查与修正”技能，并且引经据典（如国标、论文）进行了大量修正。但受限于大模型的固有缺陷，它在处理过程中产生了**文本篡改（幻觉）**和**索引计算错误（边界偏移）**。

### 4.2 关于“有没有偏移”
**存在严重的标准偏移风险。** AI 按照“学术完美标准”修正了训练集，但这很可能与官方标注团队的“实际标注标准”不一致。在比赛中，**“贴近官方标准”比“学术绝对正确”更重要**。

### 4.3 比赛策略建议

1.  **停止继续修正训练集**：你目前的 `bisai` 仓库中，AI 已经修正了前 30 块（300条）。建议停止对剩余 700 条的修正。
2.  **修复已有的格式错误**：如果你打算使用这 300 条修正数据，必须先写脚本修复那 14 条被篡改的文本，以及 108 处实体边界错误和 54 处关系锚点错误。否则这些数据是无效的。
3.  **明确赛道选择**：
    *   **如果你选 Track-A（不微调）**：不需要训练数据。你需要编写一个强大的 Prompt，从官方原始 `train.json` 中挑选几个典型的例子（Few-shot）放入 Prompt 中，直接让大模型（如 GPT-4o、Claude 3.5 等）对 `test_A.json` 进行预测。
    *   **如果你选 Track-B（微调）**：建议**直接使用官方原始的 1000 条 `train.json` 进行微调**。因为原始数据最能反映官方的标注偏好。不要使用 AI 修正过的数据，以免模型学偏。
4.  **打 A 榜的下一步动作**：
    *   读取 `test_A.json`。
    *   使用你的模型（通过 API 调用或本地推理）为每条文本生成 `entities` 和 `relations`。
    *   确保生成的 JSON 格式与官方要求完全一致（特别是 start/end 索引必须绝对准确）。
    *   打包为 `submit.zip` 并在天池提交。
