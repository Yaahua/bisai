## 一、本次守擂遇到的核心困难

### 困难 1：API 积分耗尽（最严重）

**现象：** v14 运行到一半（153条中完成56条）时，API 返回 402 Payment Required，剩余97条全部失败。

**根本原因：**
- 当前策略严重依赖大量 API 调用：v6（400次）+ v7（400次）+ v9（400次）+ v10 Gemini（400次）+ v11 nano（400次）+ v12（400次）+ v14（153次）= **约 2553 次 API 调用**
- 每次调用消耗约 0.2 积分，总消耗超过 500 积分
- 多模型集成策略本质上是"用钱买分数"，不可持续

**直接影响：**
- v14 只完成了 56/153 条补充，效果打折
- 后续无法继续迭代优化

---

### 困难 2：多模型集成导致召回率断崖

**现象：** ensemble_v3（MIN_VOTES=2，三模型）关系均值从 v9_clean 的 3.35 降到 2.16，无关系比例从 19.2% 升到 38.2%。

**根本原因：**
- 三个模型对"不确定关系"各有不同判断，取交集时大量合法关系被丢弃
- 这是多数投票策略的固有缺陷：精确率↑ 但召回率↓

**数据量化：**
- ensemble_v3 中有 153 条（38.2%）完全无关系
- 训练集期望无关系比例为 32.7%（约 131 条）
- 多丢了约 22 条有关系的句子

---

### 困难 3：VALID_TRIPLES 列表严重不完整

**现象：** 之前的后处理脚本只有 17 种合法三元组，而训练集实际有 153 种，导致大量合法关系被误删。

**直接后果：**
- v9_clean 被误删了 604 条合法关系（其中 `ABS-AFF-TRT: 106`、`CROP-HAS-TRT: 57` 等）
- 基于错误列表的所有"清洗"操作都是有害的
- 浪费了大量 API 调用来"修复"其实不需要修复的问题

---

### 困难 4：v12 的 Prompt 格式与 ensemble_v3 不兼容

**现象：** v12 的 `head_type`/`tail_type` 字段格式与 ensemble_v3 不同，导致合并脚本无法识别增量关系（新增 0 条）。

**根本原因：**
- 不同版本的脚本对关系字段的格式定义不一致
- 缺乏统一的数据格式规范

---

### 困难 5：Self-Consistency 置信度在农业文本上校准不足

**现象：** v12 的 SCIR 自纠正过于激进，将大量合法关系误判为错误并删除，导致关系均值从期望 2.80 降到 1.77。

**学术依据：** 查证文档指出，Yao et al. 在植物健康领域发现"一致性-准确性相关性在某些设置下不成立"，这在农业专业文本上尤为明显。
