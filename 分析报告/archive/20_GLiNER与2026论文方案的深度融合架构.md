# 突破召回率瓶颈：GLiNER 实体引导与 2026 顶会论文方案的深度融合架构

**作者：Manus AI**
**日期：2026-04-13**

## 1. 核心矛盾与路线 B 的战略价值

在 CCL2026-MGBIE 杂粮育种信息抽取评测（Track-A）中，我们当前的最优方案 `v18_semantic_whitelist`（得分约 0.42+）面临着一个根本性的矛盾：**多模型集成（Ensemble）极大地保证了精确率，但导致召回率严重不足（关系均值 2.26，远低于官方期望的 2.80）**。

此前我们尝试的“堆模型”策略（如引入 `gpt-4.1-nano`）失败了，因为弱模型会引入大量假阳性（FP）噪音。而纯规则白名单虽然安全，但覆盖面太窄（仅补充了 30+ 条关系）。

**路线 B（GLiNER 实体引导 + 大模型关系分类）** 的战略价值在于：它将复杂的“联合实体关系抽取（Joint NER & RE）”任务，降维成了“已知实体下的关系分类（Relation Classification given Entities）”任务。
*   **降维打击**：大模型在同时做 NER 和 RE 时容易遗漏，但如果提前告诉它“这句话里有实体 A 和实体 B”，它判断两者之间是否有关系的准确率会大幅提升。
*   **召回率保障**：GLiNER small 作为一个专门的 NER 小模型，其召回率极高。由它提供候选实体对，大模型只需做选择题，从根本上解决了大模型“看漏”实体导致的召回率损失。

## 2. 深度融合架构设计

为了最大化路线 B 的威力，我们不能仅仅是简单地串联 GLiNER 和 GLM，而是必须将其与我们在 2026 年顶会论文中调研到的前沿技术（详见报告 19）进行深度融合。

以下是设计的**三阶段融合架构**：

### 阶段一：GLiNER 高召回实体提取（基础层）
1.  **部署**：在沙盒中安装 `gliner` 库，加载 `gliner_small` 模型（CPU 推理，速度极快，内存占用小，完美契合沙盒环境）。
2.  **执行**：对测试集 400 条文本进行全量 NER 提取。为了保证召回率，可以适当调低 GLiNER 的置信度阈值（例如 `threshold=0.3`）。
3.  **输出**：每条文本获得一个高召回率的实体列表（包含实体文本、边界和类型）。

### 阶段二：基于 NLI 的 GLM 关系分类（核心层）
**理论依据**：NAACL 2025 论文 *Entangled Relations: Leveraging NLI and Meta-analysis to Enhance Biomedical Relation Extraction* [1]。
1.  **候选对生成**：对于 GLiNER 提取出的实体列表，根据官方 Schema（如 `QTL` 只能和 `TRT` 或 `CHR` 发生关系），两两组合生成所有合法的候选实体对。
2.  **NLI 范式转换**：将关系分类任务转化为自然语言推理（NLI）任务。不再问大模型“这句话里有什么关系”，而是问大模型：“已知文本 T，假设实体 A 和实体 B 之间存在关系 R，这个假设是否成立（True/False）？”
3.  **GLM-4-Flash 判决**：使用免费且上下文窗口极大的 GLM-4-Flash 模型（通过 API 调用），对每个候选假设进行 True/False 判定。

### 阶段三：不确定性感知的自一致性采样（强化层）
**理论依据**：
*   AAAI 2026 论文 *Reflect then Learn: Active Prompting for Information Extraction Guided by Introspective Confusion* [2]
*   2026 年论文 *Consistency–accuracy correlation in hard-prompted LLMs for entity and relation extraction* [3]
1.  **困惑度引导**：在阶段二中，如果 GLM 对某个假设的判定置信度较低（例如输出中包含犹豫的词汇，或者在 Prompt 中要求其输出 Confidence Score 且低于 0.8），则将该候选对标记为“高困惑度样本”。
2.  **自一致性采样（Self-Consistency）**：对于这些高困惑度样本，提高 GLM 的 Temperature（例如调至 0.7），进行 3-5 次独立重复采样。
3.  **多数投票**：在单模型内部进行多数投票。如果 5 次采样中有 3 次以上判定为 True，则最终采纳该关系。这能有效消除 GLM-4-Flash 作为较小参数量模型的随机噪音。

## 3. 预期收益与风险评估

### 3.1 预期收益
*   **大幅提升召回率**：GLiNER 保证了实体不被遗漏，NLI 范式强迫大模型审视每一个可能的实体对，这将彻底解决 `ensemble_v3` 漏标严重的问题。
*   **控制假阳性（FP）**：通过 Schema 约束生成候选对，直接排除了所有非法关系组合。同时，自一致性采样（Self-Consistency）能有效过滤掉 GLM 模型的幻觉。
*   **零成本/低成本**：GLiNER 在本地 CPU 运行（零成本），GLM-4-Flash API 完全免费，完美契合当前的资源约束。

### 3.2 潜在风险与应对策略
*   **风险 1：GLiNER 的实体边界不准**。GLiNER 提取的实体边界可能与官方标准有微小差异（例如多包含了一个冠词）。
    *   **应对**：在最终输出前，将 GLM 确认的关系与 `ensemble_v5_semantic`（语义加权投票版本）进行对齐。利用报告 19 中提到的**超串包含逻辑（Superstring-Aware Selection）**，如果 GLiNER 的实体与 `v5` 的实体存在包含关系，优先采用 `v5` 的实体边界。
*   **风险 2：GLM-4-Flash 的过度抽取倾向**。之前的测试表明 GLM 有严重的过标倾向。
    *   **应对**：NLI 范式（True/False 判定）比直接生成关系更能抑制过标。同时，在 Prompt 中加入严厉的惩罚性指令（例如：“If the relationship is merely implied but not explicitly stated, you MUST output FALSE”）。

## 4. 执行计划

1.  **环境准备**：安装 `gliner` 和 `zhipuai` 库。
2.  **脚本编写**：编写 `predict_v20_gliner_glm.py`，实现上述三阶段架构。
3.  **小批量测试**：先在测试集的前 20 条数据上运行，评估 GLM 的 True/False 判定准确率，调整 Prompt。
4.  **全量运行与融合**：全量运行 400 条数据，并将结果与 `v18_semantic_whitelist` 进行并集（OR）融合，生成最终的冲分提交文件。
